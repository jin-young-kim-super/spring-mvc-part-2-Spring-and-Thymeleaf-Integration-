public class Item {
    private Long id;
    private String itemName;
    private Integer price;
    private Integer quantity;

    public Item() {
    }

    public Item(String itemName, Integer price, Integer quantity) {
        this.itemName = itemName;
        this.price = price;
        this.quantity = quantity;
    }
}
위와 같은 핵심 도메인 모델에서 @Data 사용은 위험하다. 
@Data는 @Getter, @Setter, @RequiredArgsConstrucotr 등 많은 기능을 동시에 제공한다. 
이 말은 @Data를 쓰게 되면 예상 외의 동작을 할 가능성이 있다는 의미이다. 
좋은 아키텍처는 예상 동작 외의 동작을 원천적으로 못하게 애초에 코드 레벨에서 막는 것이 좋은 것이다.
그래서 Lombok을 사용할 때 핵심 도메인 모델에서는 필요한 기능만을 취사해서 사용해야 한다. 
그러나 예외적으로 DTO에는 @Data를 사용해도 별 상관이 없다. 


[도메인 모델은 비지니스【규칙과 행위】를 가진 객체]
・예를 들면 상품(Item)이라면 다음 같은 비즈니스 【규칙】이 존재할 수 있습니다.
ex) 상품의 가격에 대한 규칙 - “가격은 음수가 될 수 없다”라는 규칙
public void changePrice(int price) {
    if(price < 0) {
        throw new IllegalArgumentException();
    }
    this.price = price;
}
-> 그런데 만약 Item 클래스에 @Data를 붙이게 되면 아래와 같은 문제가 생길 수가 있다. 
ex) 
item.setPrice(-1000);
-> @Data는 @Setter를 지원한다. 이떄 만약 개발자가 실수로 Setter를 사용하게 되면 상품 도메인 모델의 비지니스 【규칙】을 깨버리게 되어 전체 애플리케이션 시스템이 망해 버린다. 

・도멘인 모델의 【행위】는 메서드 이름 등으로 그 행위의 의도를 드러내야 한다.
예를 들어 아래의 코드는 도메인 모델의 행위의 의도가 특정이 안 된다.
ex)
item.setQuantity(0);
이게
・재고 감소인지
(whether it is reducing stock)
・품절 처리인지
(marking as sold out)
・초기화인지
(resetting)
・버그인지
(or a bug)
알 수 없습니다.

그래서 도메인 모델의 행위의 의도는 메서드 이름으로 아래와 같이 드러내는 것이 좋다. 
item.removeStock(quantity);
item.soldOut();
이게 객체지향 설계의 원칙이다. 

DTO는 반대로 “데이터 전달”만 한다
-> 즉 단순 데이터 덩어리이며, 비지니스 규칙/행위라는 것이 존재하지 않기 때문에 위와 같이 신중히 setter 등을 사용할 필요가 없기 때문에 
@Data를 사용해서 @Getter, @Setter 등을 편히 사용해도 된다. 
※Setter는 도메인 모델에서는 되도록 사용하지 말자.
