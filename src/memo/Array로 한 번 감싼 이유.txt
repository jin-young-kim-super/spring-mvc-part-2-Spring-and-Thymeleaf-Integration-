    public List<Item> findAll() {
        return new ArrayList<>(store.values());
    }
그냥 return store.values()를 해도 되는데 굳이 ArrayList로 한 번 감싼 이유가 있다. 
결론부터 말하면, store를 오염시키지 않고 싶어서다(=store가 불변임을 담보하기 위해서이다)
findAll() 메서드는 store에 있는 모든 Item을 조회해서 화면에 보여주기 위함이다. 고로 Item 데이터에 어떠한 변경을 허락해서는 안된다.
그러나 만약에 return store.values()를 했을 때 그걸 반환 받은 쪽에서 store.put()을 통해서
store의 데이터를 변경할 가능성이 있다. 이것을 방지하기 위해서 ArrayList<>로 한 번 감싸게 되면, 반환 받은 쪽에서는 ArrayList에 뭔가 추가를 하거나 삭제를 해도 store는 불변하게 된다. 
고로, 뭔가의 불변성을 유지하기 위해서는 한번 감싸는 것이 방법 중의 하나이다. 

그럼 findById 메서드는 왜 안 감쌌냐고 의문이 들 수가 있다. 
이건 아마도 반환 받은 쪽에서 하나의 Item에 대해서 데이터를 바꾸게 허용하는 것을 허락한다는 뜻일 것이다. 
실제로 update()메서드가 존재하고 이건 Item 데이터를 변경하는 메서드이다.



- 아래 코드의 문제점
public void update(Long itemId, Item updateParam) {
    Item findItem = store.get(itemId);
    findItem.setItemName(updateParam.getItemName());
    findItem.setPrice(updateParam.getPrice());
    findItem.setQuantity(updateParam.getQuantity());
}
-> 자 이제 우리는 도메인 엔티티에는 비지니스 규칙과 행동이 있다는 것을 알고 있다.
이 점을 감안 했을 때 위 코드는 비지니스 행동에 위반되는 코드이다.
어떤 개발자가 저 set 코드만을 봤을 때, "어? itemId는 updateParam.getIt()안 해도 되나?" 혹은 실수로 setId()를 해서 데이터를
오염 시킬 수가 있다. 이것 코드에 도메인 엔티티의 비지니스 행위가 제대로 안 들어 났기 때문이다.
그래서 원래는 매개변수에 ItemUpdateDto 같은 것을 만들어 줘서 그것을 Item 도메인 엔티티에 업데이트 데이터를 매핑을 해주는 것이 정석이다.
그러면 "아 Item[Update]Dto니깐 기존의 도메인 엔티티를 갱신하는 거구나"하며 의도를 파악하여 실수를 미연에 방지 가능.
(참고로 ItemUpdateDto에는 ItemName, price, quantitiy 만이 들어 있어서야 한다.만약 IetmId가 들어 있으면 개발자는 ItemId도 갱신 대상 데이터인줄 착가할 수가 있다.)
